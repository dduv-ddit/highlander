Version 17.18 (December 2022)

New features
  • HPO terms support (see in profile tab and profile manager).
    • You can create lists of HPO terms using a built-in term browser.
    • A new profile tool can create gene lists derived from lists of HPO terms.
  • Integration of Exomiser in tools.
    • The Exomiser is a Java program that finds potential disease-causing variants from whole-exome or whole-genome sequencing data. Starting from a VCF file and a set of phenotypes encoded using the Human Phenotype Ontology (HPO) it will annotate, filter and prioritise likely causative variants. The program does this based on user-defined criteria such as a variant's predicted pathogenicity, frequency of occurrence in a population and also how closely the given phenotype matches the known phenotype of diseased genes from human and model organism data.
    • Note that you can use Exomiser in two ways:
      • Sample mode: each sample is analyzed separately, but you can run a batch of samples at once with the same HPO terms.
      • Family mode: you analyze a family as a whole (specifying the pedigree), but one at a time.
    • After that select your samples or families.
    • For each one, you'll see if you already have Exomiser results or if you need to run it.
    • If you have to run it, it will be on the cluster, not on your computer. You'll get an email when results are ready.
  • Integration of Kraken in tools.
    • Kraken is a system for assigning taxonomic labels to short DNA sequences.
    • Select your samples.
    • For each one, you'll see if you already have Kraken results or if you need to run it.
    • If you have to run it, it will be on the cluster, not on your computer. You'll get an email when results are ready.
  • HGMD importation tool.
    • HGMD license doesn't allow Highlander to have their annotations in columns, but they can be saved in public comments.
    • To import HGMD annotations for a set of variants, proceed as follow:
    • Export your Highlander results as a VCF (using export to VCF tool in Highlander)
    • Submit that VCF to HGMD as batch results
    • Save the batch results as an HTML file (right click outside the table in your browser, and hit save as...)
    • Use this tool to import the HTML file in Highlander
    • HGMD annotations for those variants will be visible in Highlander public comments, for any matching variant
  • Gnomad internal annotation databases support.
    • Gnomad frequencies are not anymore retreived from DBNSFP.
    • It means you will now see the gnomad frequency for ANY variant having one (even if it's not a non-synonymous coding SNV or an INDEL).


Version 17 (November 2020)

New features
  • Multiple reference genomes support. Each analysis is now linked to a specific reference.
  • Variants are now diffenriated by gene. It means that if a variant is on a genomic position overlapped by 2 genes, you will see two lines in the Highlander table (with different annotations for each gene). Note that VCF must be reimported to see the change in existing samples.
  • Change and gene statistics overhaul
    • Variant count is replaced by allele count (like gnomad, gonl, 1000G, etc).
    • An allele present in multiple samples from the same individual is only counted once.
    • 'Change statistics' category becomes 'Allele frequencies'. All related fields have been replaced by new ones (but most old fields have a correspondance).
    • New fields are local_af (allele frequency), local_ac (allele count) and local_an (allele number), plus local_het and local_hom (count of heterozygous and homozygous).
    • A field local_pathologies gives the number of pathologies have at least one allele with the variant.
    • Same fields exist for germline and somatic alone.
    • New magic filter 'Allele frequency in a pathology' where you can select a pathology and a frequency filter criterion on that pathology alone.
    • The fields giving number of variant per pathologie have been removed. Instead you can display a table with a new detail box, or use a magic filter to combine allele frequencies per pathology.
    • There is no more differenciation between 'filt' and 'unfilt' (i.e. only unfilt has been kept).
    • Fields 'change_num_xxx_xxx_groups' and 'change_num_xxx_xxx_details_groups' have been removed. Information is still accessible in a detail box.
    • 'Gene statistics' category has been removed, with all its related fields.
    • All detail boxes named 'Same change (xxx)' and 'Same gene (xxx)' have been removed.
    • A new detail box 'Allele Frequency per pathology (details)' gives you the same information than before in one box, with each individual presenting the allele (and the list of sample between parenthesis). You can select all samples, or only germline/somatic.
    • A new detail box 'Allele Frequency per pathology (table)' gives that information in a summary table (showing only pathologies that have at least 1 allele).
    • Filters saved in your profile have been modified to fit with the new fields, but some of them could still need to be updated by hand.
  • User annotations
    • 5 new fields have been added, on which you can filter. They are called num_evaluated_as_type_X (X = type I to V), and gives the number of samples with this variant already evaluated to X.
    • Private and public annotation on a variant (of_interest flag and public/private comments) are now saved for all samples. e.g.Before, putting a comment for a variant x selected on patient y was limited to that patient. Now clicking on the same variant on patient z will show the same comment.
    • Private and public comments can now be added for samples.
    • There is a new flag in variant evaluation, called 'Reporting'. It can be set to YES/NO if the variant has been reported to the patient, or should not be.
    • The detail box 'Samples with same evaluated variant' has been duplicated : one detailled (like before) and another with only the total count. A button allows to show only evaluations (removing all the NOT_CHECKED status from table for more visibility).
    • Date of evaluation has been added to the fields and the detail boxes 'Samples with same evaluated variant'.
  • List of values are now linked to a specific field.
    • Encoding dialog colors unknow values in red.
    • Encoding dialog has a button to select values from the database (instead of a separate dialog).
    • In profile manager, lists are separated by fields. Only fields for which you have lists are shown. If you want to create a list for a new field, select the 'List of values' node.
    • For your existing list, field has been guessed depending on the values. If the field wasn't recognized, it been set to 'unknown' by default. You can change it in the profile manager.
    • You can now add comments for each value of your lists (e.g. why you added this gene to that list).
  • List of intervals are now linked to a specific reference.
    • In profile manager, lists are separated by reference. Only references for which you have lists are shown. If you want to create a list for a new reference, select the 'List of intervals' node.
    • When a list of interval is needed for a specific analysis, it shown all lists with a compatible reference. For example, GRCh37 and hg19 are compatible.
    • For your existing list, reference has been set to GRCh37. You can change it in the profile manager.
  • New external links: Variant in Varsome, Variant in ClinVar, Variant in Franklin, Protein in UniProt, Gene in GTEx.
  • Administration tools
    • Project management > projects: you have new fields for family, individual, index_case, population, trimming, duplicate removal and normal sample (e.g. for MuTect). Reference has been moved at the analysis level.
    • Project management > samples: new tab to manage all samples at once, instead of per project.
    • Database management > Analyses: completely revamped. You can set a reference genome and sequencing target for each analysis.
    • Database management > References: new tab to manage reference genomes. Each reference can be linked to a set of annotation databases.
    • Database management > Populations: new tab to manage populations. Each individual can now be linked to a population defined there.
    • Database management > Fields: new tab to manage database fields. Fields description, source and annotation scheme can be changed here, as well as new custom fields and visibility per analysis.
    • Database management > Reports: new tab to manage files generated by pipeline and useful in Highlander. You can add software and their files there so they can be downloaded from Highlander (Tools > File downloader).
    • Client management > Global settings: new tab to manage settings that were in settings.xml file. All settings present there do no longer need to be in the settings.xml of each client (they are managed here for all client, like proxy settings). Note that those settings can still be defined in settings.xml, and they will replace global settings for this particular client.
    • Client management > External links: new tab to manage 'External links' detail box available in Highlander. All buttons that should display in the detail box can now be configured here, and you can easily add new ones (or remove/disable the ones you don't want).
    • User management > Management: a table of all users, can be use to modify user information. Users can now be put in groups, and be deactivated (they can't login anymore, but their data is kept).
    • User management > Mail users: new tab where you can write an email and send it to all of your Highlander users.
  • Variant importation
    • Structural variants annotated by AnnotSV can now be imported.
    • When a variant span multiple genes, it is now split in multiple variants, with same chr/pos/length/ref/alt but different gene_symbol. All fields for which the annotation depends on the gene are specific to the gene_symbol of the variant.
    • When choosing which genes are affected by the variant, only the genes effectively present at the variant positions are taken into account (from 5' to 3' UTR included). SnpEff can gives "affected" genes up to 5000 nucleotides of distance of its UTR, they are not taken into account anymore.
    • Field gene_number is now reflecting the real number of genes at that positions (before it was based on the number of genes selected by snpEff).
    • Variants like CTTT > CTT in the VCF (because of multiple alternatives) are now reduced to CT > C, which is the correct notation.
    • Fields specific to a variant caller (e.g. Mutect, Ion torrent) can be now be specific to an analysis (see Administration tools for custom fields).
    • VCF parsing informations can now be changed / added in the administration tools.
  • Update of all annotation sources
    • Last version of Ensembl (for GRCh38, GRCh37 has been frozen since Ensembl 75)
    • DBNSFP 4.1
    • ~400 new annotations from DBNSFP
    • Consensus prediction has changed, see column description for details. Now uses 20 different sofware for missense variants.
    • Allele frequency in population now has exac and gnomad detailled columns (per population, and specific datasets like non-TCGA or non-psych).
    • Cosmic 92 dataset
    • Whole new gene annotations category of columns. Contains data found only in "gene annotations" detail box (gene description, pathways, etc), but also lot of new annotations (like pLI scores from ExAC).
    • Some annotations have disappeared: ARIC5606_AA_AF, ARIC5606_EA_AF, ESP6500_AA_AF, ESP6500_EA_AF, the1000G_AF, consensus_MAF, ARIC5606_AA_AC, ARIC5606_EA_AC, consensus_MAC, the1000G_AC, transcript_uniprot_acc, clinvar_rs, cosmic_count, cosmic_id, unisnp_ids, lof_tolerant_or_recessive_gene, exac_an.
  • New detail boxes
    • Predictions spider chart now uses DBNSFP (converted) rank scores for all available predictors. In general rank scores are computed this way: predictor X scores were ranked among all predictor X scores in dbNSFP. The rankscore is the ratio of the rank the predictor X score over the total number of predictor X scores in dbNSFP. If there are multiple scores, only the most damaging (largest) rankscore is presented. The rankscores are always between 0 and 1 (but min/max score can be more/less).
    • Conservation spider chart. Similar to the existing spider chart for predictions, but using all conservation (converted) rank scores from DBNSFP.
    • Variant in family. Displays all samples from each individual of the whole family, and indicates if the variant is found in other samples, whichever the analysis.
    • Alignment. You can now open a menu with new feature by clicking anywhere on the alignment. You copy read information, sequence or launch a BLAT on it.
    • Detail boxes order and visibility is now saved per analysis (you can have a different ordering for each analysis). You can also reset the ordering with a button.
    • You can search fields in detail boxes. Field will be highlighted in the detail box, and a button allows to scroll vertically to the detail box containing it, and scroll horizontally to the column in the main table.

Version 16 (August 2019)

New features
  • Better consistency for gene symbols. Depending on the source (HGNC, UCSC, Ensembl, RefSeq, ...) and the version, canonical gene symbols could be different, leading to problems when comparing annotation in Highlander, coverage tool, alignment, IGV and user gene lists. To minimize those discrepancies, some changes and tools have been implemented.
    • Ensembl is now the only source of gene data used within Highlander. The same Ensembl version is used everywhere. So be aware of the Ensembl version installed on the Highlander server, it could still show differences with older/newer versions. e.g.If GRCh37 is used, the last version of Ensembl that supports it is 75 (Feb 2014). If you compare to a more recent GRCh38 version, some gene symbols have dissapeared, appeared, or changed their name.
    • Coverage tool: Highlander can generate a gene list file to use with GATK DepthOfCoverage (used for coverage tool in Highlander). GATK uses UCSC export by default, so Highlander can generate a similar file based on the Ensembl version installed. Using that file in the pipeline should align coverage tool gene symbol with everything else.
    • Alignments: now uses Ensembl (see next point). Highlander was using UCSC to draw genes in alignments, now it should match variant annotations and be far more complete.
    • Variant annotation: it hasn't changed, it was already Ensembl used before.
    • User gene lists: when you encode a gene list, you can now validate it. If a gene symbol is not found in Ensembl canonical list but as a synonym in HGNC, Highlander will propose to replace it. The button is present in the 'profile' tab and in all windows where you can encode gene symbols.
    • Note that, by extension, the validation tool works for other fields like patient or pathology. Fields with real and natural numbers, or infinite possible text values are not available in the validation tool.
    • IGV: IGV still uses UCSC, so you could find discrepancies with Highlander. Alignments tools (and new Alignment Viewer) available within Highlander should be used instead.
  • Alignment visualization box has improved:
    • Zoom feature.
    • Navigation buttons move of a number of bases equal to half the total bases displayed.
    • Button to 'frame shift' all amino acids is now completely based on selected variant and only reflects this shift, accurately.
    • Display a tooltip with mapping quality when hovering reads, and genomic position when hovering reference.
    • Size of deletions greater than 1 are indicated inside the deletion.
    • Use of Ensembl for gene data (instead of UCSC, see previous point). More entities are now showed in different colors (processed transcript, RNAs, pseudogenes). Region displayed should now always match annotations (in precedent versions, discrepancies could arise between transcript used in annotation -Ensembl canonical transcript- and alignment -RefSeq NM transcript-).
    • Gene mouse-over displays more info (gene symbol, exon/intron rank, UTR/CDS, Ensembl gene and transcript ids, RefSeq NM id and transcript biotype).
    • An 'alignment viewer' is available in the tools tab. You can use it to quickly view a given position (by default corresponding to the selected variant) in different samples. You can also "pin" samples for a multiple alignment view. Selecting multiple variants from a filter result will open one viewer per unique variant, and select/pin all patients having the variant from those results. A second button in the tools tab (with a 'pin' icon) allows to preselect a list of patients to pin. It can be handy when you check 'de novo' variants between samples (like with magic filters sample-specific or combined heterozygous), or with trio's.
  • External links box: if multiple variants are selected, buttons will open web sites for all uniques genes/variants/etc. e.g. If 10 variants are selected and 3 are uniques, clicking on ExAC button will open 3 tabs in your browser, one for each unique variant. [before: only the 1st variant of the selection was taken into account]
  • Columns transcript_refseq_mrna and transcript_refseq_prot now give version number. e.g. NM_005529 is now noted NM_005529.5. Note that old annotations will not be updated, samples must be reimported.
  • PedigreeChecker has tooltips for each cell of the table. Tooltips for gender column give the exact hom/het ratio.
  • Fitlers templates: several bug corrections (folders were not sorted/searchable, lower case patient names caused problems).

Version 14 (November 2015)

New features
  • New 'alignment' detail box: a complete 'IGV-like' view of the read 40bp around the selected variant.
    • Informations are directly read from the BAM file, so it must be accessible for the detail box to work.
    • Reads are ordered with selected variant alternative first, other alternatives following and ending with reference.
    • Reads and bases use the same color code than IGV : forward reads in pink, reverse reads in blue, A in green, C in blue, G in orange, T in red and insertions in violet.
    • Reads and mismatching bases are shaded depending on their mapping quality (MQV < 60 are shaded) and base phred quality (QV < 30 are shaded).
    • Hovering over reference position of the variant show detailled read composition for this position
    • Hovering over refseq track gives you gene symbol and exon number
    • Hovering over an insertion of mismatching base gives you nucleotides and their phred score
  • Better handling of filters combination
    • There is a new 'kind' of filter: 'Combo Filter'.
    • Now, you only create (and save to your profile) one Combo Filter.
    • A Combo Filter is either
      • a Custom Filter (one or more basic filtering criteria, like 'patient = X AND gene_symbol = TEK,SOX18')
      • a Magic Filter
      • two or more Combo Filters combined with AND (intersection) or OR (union).
    • Custom Filter criteria combined with AND/OR are treated as usual, i.e. intersection/union of database record, meaning that creating a filter like 'patient = X AND patient = Y' returns nothing (no record can belong to patient X AND Y).
    • Magic Filters are combined with AND/OR at the variant level. It means that combining 'Variants common to patient W and X' AND 'Variants found in Y and not in Z' could give you something (variant with same chr-pos-ref-alt found in both filters, despite their other fields value).
    • The interface has been changed accordingly. The main 'add Custom/Magic/Loaded from profile filter' buttons add a new filter in the main Combo Filter. Small + signs in each filter allow to add a new level of filters (Custom, Magic, Loaded from profile) combined with AND/OR.
    • This new paradigm allows to create a broader range of queries, and to use different Custom Filters associate to different Magic Filter. For example, you could combine ['Variants common to patient W and X' AND 'consensus_MAF < 1%'] AND ['Variants found in Y and not in Z' AND 'consensus_MAF < 20%'], which was not possible before.
    • In the 'Sample specific variants' Magic Filter: you can now ask for the intersection of variants to keep, in addition of the union. It should allow to find variants following models like dominant, X-linked or imprinting.
    • In the 'Common-gene variants' Magic filter: you can now find genes affected by AT MOST or AT LEAST X variants in a certain number of patient.
    • In the 'Genomic intervals' Magic filter: you can now keep variant inside (default) or outside the given intervals.
    • IMPORTANT: Note that most of your existing saved filters won't probably work anymore. Try to load each, and if Highlander throws an error message for one, please delete it and recreate it using the new interface.
  • New interface for filter creation
    • A new button in the filter panel opens a window with a tree-like view of the filter.
    • Expand/collpase nodes to show more/less detail on each subfilter.
    • Buttons change depending on the selected node, corresponding to options available from the selected filter.
    • When a specific node is selected, adding a new subfilter with the buttons will add it as a part of the selected filter.
  • New profile management interface
    • All profile management buttons of the profile panel have been removed, and replaced by a single button that open a tree-like view of your profile.
    • Folders can now be created to organize any category.
    • Elements are now only saved for the current analysis, but you can duplicate them for other analysis in the profile manager.
    • Elements can be moved to other folders by dragging them with the mouse.
    • Other operations are available through the buttons : rename, edit, delete, duplicate, share.
    • The right panel of the manager show a complete description of the selected element.
  • New gene annotation options
    • In private annotations, you can now write comments about the gene of the selected variant. The comments will be associated to ALL variants found in this gene, for any patient. You are the only who can see or change thoses comments. Note that you can set comment for multiple genes at once if more than one are selected (you only see the comments of the first one), so be careful.
    • In private annotations, you can mark a gene 'of_interest' like you mark a variant. The flag will be associated to ALL variants found in this gene, for any patient. You are the only who can see or change this flag. Note that you can set the flag for multiple genes at once if more than one are selected (you only see the flag value of the first one), so be careful.
    • In public annotations, you can now write comments about the gene of the selected variant. The comments will be associated to ALL variants found in this gene, for any patient. Anyone can see or change thoses comments.
    • Field 'of_interest' changed to 'of_interest_variant', and 'public_comments' changed to 'public_comments_variant'. New gene associated fields are 'of_interest_gene', 'private_comments_gene' and 'public_comments_gene'.
    • In public comments (variants and genes), Highlander automatically adds who has made the last modification and when.
    • You can now share private comments about selected variant or gene with another user. Use the share button on the right of the comment box, choose another user and he or she will have your comment with your name in addition of his/her existing comments. Note that you can't share more than one comment at a time, even if you select more than one variant/gene.
  • A new button on top of detail boxes allows to bring a selected one on top of the list.
  • New button in Navigation panel to hide rows based on a selection of patients.
  • New button in Profile panel (and Profile manager, with edit option too) to create lists of values by choosing from database fields (to easily create/edit a list of genes of patients).
  • External links detail box: add an Decipher button.
  • When the horizontal blue arrows appear to scroll (like with long filters), you can now right click on them to scroll directly to end/start.
  • You can open BAM files from multiple analysis at once in IGV.
  • New search bar in the run statistics details.
  • In the run statistics charts, possibility to separate SNPs from INDELs.

    • VEST 3.0: vest_score field added. Score ranges from 0 to 1. The larger the score the more likely the mutation may cause functional change. In case there are multiple scores for the same variant, the largest score (most damaging) is presented. Please refer to Carter et al., (2013) BMC Genomics. 14(3) 1-16 for details.Please note this score is free for non-commercial use. For more details please refer to http://wiki.chasmsoftware.org/index.php/SoftwareLicense. Commercial users should contact the Johns Hopkins Technology Transfer office.
    • CADD: cadd_raw and cadd_phred fields added. CADD raw score for funtional prediction of a SNP. Raw CADD scores come straight from the model, and are interpretable as the extent to which the annotation profile for a given variant suggests that that variant is likely to be observed (negative values) vs simulated (positive values). These values have no absolute unit of meaning and are incomparable across distinct annotation combinations, training sets, or model parameters. However, raw values do have relative meaning, with higher values indicating that a variant is more likely to be simulated (or not observed) and therefore more likely to have deleterious effects. On the other hand CADD phred-like score (-10*log10(rank/total)) are based on whole genomeCADD raw scores. A scaled C-score of greater of equal 10 indicates that these are predicted to be the 10% most deleterious substitutions that you can do to the human genome, a score of greater or equal 20 indicates the 1% most deleterious and so on. If you would like to apply a cutoff on deleteriousness, you should put it somewhere between 10 and 20 (15). Please refer to Kircher et al.(2014) Nature Genetics 46(3):310-5 for details. The larger the score the more likelythe SNP has damaging effect. Please note the following copyright statement for CADD: CADD scores (http://cadd.gs.washington.edu/) are Copyright 2013 University of Washington and Hudson-Alpha Institute for Biotechnology (all rights reserved) but are freely available for all academic, non-commercial applications. For commercial licensing information contact Jennifer McCullar (mccullaj@uw.edu).
    • PhyloP: phylop field removed and fields phyloP46way_primate, phyloP46way_placental and phyloP100way_vertebrate added. phyloP (phylogenetic p-values) conservation score based on the multiple alignments of 10 primate / 33 placental mammal / 100 vertebrate genomes (including human). The larger the score, the more conserved the site.
    • phastCons: fields phastCons46way_primate, phastCons46way_placental and phastCons100way_vertebrate added. phastCons conservation score based on the multiple alignments of 10 primate / 33 placental mammal / 100 vertebrate genomes (including human). The larger the score, the more conserved the site.
    • ARIC5606: fields ARIC5606_AA_AC, ARIC5606_AA_AF, ARIC5606_EA_AC and ARIC5606_EA_AF added. Alternative allele counts/frequency in 2403 exomes of African/European Americans from the Atherosclerosis Risk in Communities Study (ARIC) cohort study.

    • Splicing predictions: fields is_scSNV_RefSeq, is_scSNV_Ensembl, splicing_ada_score, splicing_ada_pred, splicing_rf_score and splicing_rf_pred added. The is_scSNV fields indicate whether the SNV is a scSNV according to RefSeq/Ensembl (i.e. in splicing consensus regions, -3 to +8 at the 5’ splice site and -12 to +2 at the 3’ splice site). Version 0.8 (April 2013)

New features
  • BAM Viewer (in toolbar Tools): allow to visualize the selected position in details in ALL available BAM files (table exportable in Excel)
  • Brand-new right panel with detailled info on the selected variant
  • Information is splitted amongst different boxes, each one can be expanded or collapsed (using the +/- button), and you can reorder boxes using drag and drop
  • The boxes position and visibility (expanded or not) is automatically saved in your profile, so you don't have to reorder it all the time !
  • You have one box for each column category, so you can see all information on a variant even if you have selected only a few columns for your query
  • 'Error detection' shows 'WARNING: this variant is detected as an error' in red or 'This variant is NOT detected as an error' in green
  • For Change and Gene statistics, you have extra boxes that give you the exclusive list of samples in each category
  • 'Polyphen prediction for other transcripts' shows Polyphen prediction for all other 'non-canonical' transcripts (the canonical transcript is used in the table and is NOT repeated here)
  • 'SnpEff prediction for other transcripts' shows SnpEff prediction for all other 'non-canonical' transcripts (the canonical transcript is used in the table and is NOT repeated here)
  • 'Public annotations' allows you to set the fields check_insilico, check_validated_change, check_somatic_change and public_comments.
    • check_insilico: Any user who has checked this variant insilico (e.g. with IGV) can set this flag to true if it's likely a real change, or to false if it's likely a sequencing error (e.g. alternative allele not specific to pathology)
    • check_validated_change: Any user who has checked this variant can set this flag to true if the change has been confirmed in the lab, or to false otherwise
    • check_somatic_change: Any user who has checked this variant can set this flag to true if it's a confirmed somatic change, or to false if it's a germline change
    Once any of these fields are set to true or false (instead of 'no value') by any user, only this user (or an administrator) can change it again. The public_comments can be updated any time by anyone, so please avoid to erase other users comments, just add yours. Note that any change made in this box is automatically updated in the database, but NOT in your table, you need to re-send your query to see the columns change.
  • 'Private annotations' allows you to set the fields of_interest and private_comments. Those fields are user-specific and only visible by you, so if you mark a variant of_interest, it's only for you and it will only affect your queries. It's the same for private_comments, no other user can see them.
  • 'Gene and sample coverage' shows the vertical and horizontal mean coverage for the selected gene in the selected sample, the gehu group and the whole database
  • 'Run statistics' shows all information and statistics relative to the run that yield the selected variant
  • 'External links' shows buttons that allow you to directly open your web browser with your variant/gene entered in different web sites like dbSNP, UCSC, Ensembl, OMIM, PubMed, NCBI, Entrez or HGNC. The variant is also automatically submitted to Mutalyzer position converter and one button is available for each result, showing the transcript id and cDNA positon and allowing to submit it to Mutalyzer name checker
Bugs correction
  • Highlander asks for saving your filter even if it's a new one (before it had to be loaded)
  • Decimal coma in numbers are automatically changed to decimal point for filters (decimal coma is not recognized by the database)
  • There were still database disconnection problems (for queries that update the database), they should be fixed now
  • Add scroll bars in the magic filter creation dialogs, in case of large patient list


Version 0.7 (March 2013)

New features
  • New magic filter Variants common to samples: keeps only the variants common to a set of samples. In addition, for each sample, you can set a (possibly different) value for some fields: zygosity, allelic depth and allelic depth proportion. For example, you could obtain the common variants in a family, heterozygous in the parents and homozygous in the child.
  • New magic filter Common-gene variants: Keep only variants found in `common` genes (i.e. genes that have at least one variant in all given samples). In addition, you can set dbsnp_flagged, consensus_prediction and filters criteria.
  • New magic filter Combined-heterozygous variants: Given a trio (2 parents and their child), find all potential combined-heterozygous variants. Each of the variants found in the child, should be present in only one of the parents, and at least two such variants should be present in the same gene.
  • In the filters tab, you now have a count button to evaluate the number of variants that will be retreive by your query
  • You can now modify a filtering criterion (custom or magic) by double-clicking it
  • If you create a filter without saving it, or modify an existing fitler, you will be asked if you want to save the changes when you exit Highlander
  • The IGV button has new features: you can include dbsnp tracks (132, 135 and 137) and tracks with called variants of selected patients
  • A new IGV button in the 'tools tab' allows to jump to the chromosome and position of the selected variant in IGV, if it's open inside Highlander
  • Highlander now informs you about the database load, using a colored ball in the bottom left bar. A green ball means that database is idle and ready for use (no query is currently being processed). A pink ball means that a query is currently being processed, and the username of the first query in the queue is also shown, so you know if it's yours (if not, your query could run, but slowly). An orange ball means that a set of new samples is being processed by Highlander pipeline on the cluster: the database is not yet updated, but will be in the following hours, and data integrity is not assured (e.g. the new samples could have already been inserted in the database, but the statistics are not yet updated). A red ball means that the database is currently updated, so please don't launch queries during that time (anyway, any queries will probably take forever to finish).
  • An email is sent to you when samples of interest for you are imported into the database and available in Highlander
Miscellaneous
  • When you create masks, only selected column are shown, in the order you haven chosen
  • You have now two different buttons for possible values from the database or from the actual table
  • Filter criteria are now totally case insensitive. It will simplify your life for a lot of fields (sample_type, filters, etc), and is only problematic for old_codon and new_codon fields (e.g. searching for cGt gives you variant with cGt, Cgt and cgT)
  • Queries with only statistics are no more allowed. To ensure that the entire database is not downloaded to your computer you must either: include at least one non-statistic field in your custom filter (patient or gehu_group are good candidates), or include at least one magic filter
  • The magic filter Familial relationships and zigosity has been removed, but it's functionality can be achieved with the new filter Variants common to samples (see in New features)
  • Statistics field have now a better description
  • The memory bar now indicates the total amount of memory currently allowed, and is expressed in Gb
  • Better MacOs X integration (like about, preference and quit in the menu, cut/copy/paste using command and not ctrl)
  • Current date and time is printed with each query and error in the log
Performances
  • Asking from possible values from the database should be nearly immediate
  • Queries containing magic filter are automatically limited to the patients selected in the magic filters (should be faster, and you don't need anymore to add a custom criterion with your patients)
Bugs correction
  • Version 0.7.1 correct some bugs with IGV (don't move to the right position after closing IGV, changing Highlander icon, patient selection) and with database load icons
  • Corrected some database possible values list for some statistics fields that were incorrect (X_num_Y_Z fields)
  • Highlander should reconnect to the database instead of crashing if you left it inactive 8 hours or more
  • Sorting and possible values now work correctly even you have masked columns
  • It's no more possible to sort a masked column, and if you mask a sorted column (or load a filter with an masked or inexistant column), it will be remove from the sorting criteria.
  • IGV window can now be closed and reopened without problems. Bonus, the tracks should now fit to the window and you can open IGV even if no patient is selected !
  • You get an error message if you run a query that uses a value list from your profile that doesn't exist (or was deleted/renamed/emptyied)
  • When you click on cancel when asked for a column selection / mask name, you go back to the column selection dialog
  • When you edit the active column selection / mask in your profile, it automatically refresh the table to reflect the changes
